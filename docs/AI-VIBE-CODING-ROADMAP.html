ITHub 自助服务门户

一个面向终端用户的 IT 自助 Demo —— 登录、对 AI 助手说话、按一下"转人工"就生成工单，ITHub admin 后台可以全程追踪。从立项到上线 4 天，97 次提交，全程由 Claude Code 协作完成。

📅 2026-06-27 → 2026-06-30 📦 97 次提交 🧑‍💻 Vite + React + Express 🌐 GH Pages + Render 🔗 线上 Demo

00TL;DR

终端用户（员工）打开这个 Portal，可以：

  1. 用 ITHub 账号登录；
  2. 和一个接入了公司知识库的 AI 助手对话（基于 RAG：先在 KB 里搜相关文章塞进 prompt）；
  3. 对话解决不了就点"转人工"——MiniMax 把整段对话压成一句话描述，自动建工单 + 同步完整聊天记录到 ITHub Journals；
  4. 在"我的工单"看进度、查时间线、加备注；
  5. 工单关单后还能"总结为 KB 草稿"，让 AI 帮写好让人 review 再发布到 ITHub KB。

Demo 跑在 GitHub Pages（静态前端）+ Render（Express 后端代理），后端是 唯一 持有 ITHub AccessToken 的进程 —— 浏览器从头到尾摸不到真实凭证。

04API 获取方法论（重点章节）

这个项目最值钱的部分 —— 没有任何公开 API 文档。ITHub 不像 GitHub/Stripe 有 OpenAPI，全靠探针摸索。把方法论沉淀成 4 步：

Step 1 · 列出端点猜测

从 UI 操作倒推：用户能看到 KB 文章 → 至少要 GET /api/Knowledge/KnowledgeArticles；用户能改状态 → 至少要 PUT /api/Knowledge/KnowledgeArticles/{id}。把这些猜的端点写到一个清单。

Step 2 · 写探针路由，把每个猜测打一遍

在 server/src/routes/_debug/ithub-kb-publish.ts 里写了独立的 POST /_debug/ithub-kb-publish/listProbe、putProbe、fillProbeV1..V5。每个探针只做一件事：打一个端点，把响应原样 dump 出来。

// 探针示例：fillProbeV5（Content-Type 修好之后的版本） POST /api/Knowledge/KnowledgeArticles/{id} Content-Type: application/json AccessToken: eyJ... CustomerTag: ciscoinnovation1 { "Identifier": "K100103", "CustomerId": 3, "CustomerTag": "ciscoinnovation1", "KnowledgeBaseId": 1, "ParentKnowledgeCategoryId": 4, "KnowledgeCategoryId": 5, "KnowledgeCategoryName": "Hardware", "Summary": "…", "DescriptionText": "…" ← ❌ 被静默丢弃！ "Description": "…" ← ✅ 实际生效的字段 "KnowledgeArticleStatus": 0, "Active": true, "AccessFlags": 2147483647, "KnowledgeArticleAccessFlags": 2147483647, "KnowledgeArticleServiceDeskAccessFlags": 2147483647 } ← 200 Body: true // ITHub 不返回对象，只返 true

Step 3 · 沉淀状态码矩阵

把每次探针的 HTTP 状态码记到一个表里。命中率最高的发现：

状态码触发条件我们的处理
200 / 204成功（PUT 的成功 body 是 true）正常
400缺 NOT NULL 字段 / 字段类型错补字段，重试
401AccessToken 过期 / CustomerTag 不匹配重新登录
404路径错（ITHub 有"必须在顶层"的奇怪规矩）改路径
405用了 PATCH / DELETE（PUT 是唯一能改的动词）改用 PUT 全量覆盖
415Content-Type 错（很多老 endpoint 期望 form 不是 json）改 Content-Type 重试

Step 4 · 探针沉淀成生产代码

当一个探针跑出 200，就把它的逻辑搬到正式路由里。探针本身删掉（commit 18e14a1 一次性清理 5 个 fillProbe + 临时 helper）。

这个方法论学到的 3 件事

  1. ITHub 写操作没有 PATCH —— 凡是更新都是 PUT 全量。代码里要 "读最新 → 改一两个字段 → 整体 PUT 回去"，不能假设 PATCH 会合并。
  2. 字段名靠注释和探针双确认 —— 我有两次猜错字段名（DescriptionText → Description、Pending → Published）。TypeScript 类型 + 探针响应双验，是最稳的。
  3. 读副本延迟是真的 —— admin 后台改了状态，立刻 GET 拿到旧值（ModifiedUtc 还是 5 秒前的）。所有写后读都等 2-3 秒，必要时多次重试。

05安全模型：代理模式

整个项目的安全核心就一句话：浏览器永远摸不到 AccessToken。

为什么必须代理

如果前端直接打 ITHub：

  • AccessToken 必然进 JS bundle，任何人 DevTools 能直接复制
  • CustomerTag / ApiKey 同理
  • CORS 要在 ITHub 上配白名单，跨域 cookie 又是一堆 SameSite 的坑

代理模式把这些凭证全部锁在服务端，浏览器只认 HttpOnly cookie。

请求流（登录一次，后续都是这样）

浏览器
fetch /api/chat/init 带 cookie: sid=…
Render (Express)
requireSession 拿 sid → 查内存 session → 拿到 AccessToken
ITHub 上游
注入 AccessToken + CustomerTag + 20s 超时

登录那一刻

浏览器
POST /api/auth/login { identity, password }
Render
调 POST /api/Security/AccessTokens 拿 token
生成 sid，存 Map<sid, {accessToken, customerTag, ...}>
Set-Cookie: sid=…; HttpOnly; SameSite=None; Secure
ITHub
返回 AccessToken 字符串

代码层

  • server/src/session/middleware.ts — requireSession 守卫所有业务路由（除了 /api/auth/login、/api/health）
  • server/src/http/ithubClient.ts — 唯一接触 ITHub 上游的地方。新端点只能在这里或它的调用方出现
  • Session 默认 8h 过期，每 5min 清理一次过期项
  • 跨站部署（GH Pages → Render）强制 SameSite=None; Secure（hasExternalOrigin 判定）

08踩坑清单（按严重度排序）

所有坑都是真金白银踩出来的。每条都是 "症状 → 根因 → 修复"。

① KB body 字段是 Description，不是 DescriptionText

P0
症状
调用 PUT /KnowledgeArticles/{id} 返回 200，但 ITHub admin 后台文章正文是空的
根因
ITHub 不返回 PATCH 语义错误，只是"字段名不匹配被静默丢弃"
修复
把 body 里的 DescriptionText 改成 Description。需要把 5 个 fillProbe 跑一遍才能确定

② ITHub PUT 是全 SQL UPDATE，不是 PATCH

P0
症状
只 PUT 一个字段，ITHub 把其他字段全清成 NULL
根因
ITHub 没有 partial update 概念（PATCH 永远 405）
修复
在 kbRepair 里 "GET 当前完整对象 → 改 Summary/Description/Status → 整体 PUT 回去"

P1
症状
/KnowledgeBases/{kbId}/KnowledgeArticles?$top=100 只返回 10 篇，$top 被忽略
根因
嵌套路径强制分页 10，$top 不生效
修复
改用顶层 /KnowledgeArticles?KnowledgeBaseId=... —— 一次拿全 21 篇

④ 工单创建路径：/ServiceDesk/Tickets 永远 404

P1
症状
POST /api/ServiceDesk/Tickets 报 "The resource cannot be found"
根因
ITHub 工单实际在 Nms 控制器下，ServiceDesk 控制器只管 auto-ticketing profile
修复
读用 /Nms/Tickets；创建用 /ServiceDesk/Customers/{tag}/TicketTemplates/{tid}/TicketIncidents（必须 customer-scoped + ApiKey）

⑤ 跨站 session cookie 被浏览器拒收

P1
症状
本地登录 OK；推到 GH Pages 后每次请求都 "未登录"
根因
GH Pages (https://leo...) → Render (https://ithub...) 跨站，浏览器拒绝 SameSite=Lax 的 cookie
修复
加 hasExternalOrigin 判定；跨站时强制 SameSite=None; Secure

⑥ React error #31："Objects are not valid as a React child"

P1
症状
点开 K100105 整页白屏，console 报 React error #31，对象 {SecurityContextType, ItemId, Type, Name, CustomerTag, ...} 被直接当子节点渲染
根因
ITHub 对部分字段（KnowledgeCategoryName / CreatedBy / ModifiedBy）返回 rich 子对象而不是字符串，按文章/租户有差异
修复
在 KbPage 加 safeStr(v) 防御：对象 → 抽 .Name / .UserName / .DisplayName

⑦ HTML 实体双重编码 + hex entity

P2
症状
文章正文显示成 查询（"查询"的 hex 编码）
根因
ITHub 返回的内容已经被双重 HTML 转义，stripHtml 只解了 &#(\d+);（十进制），没解 &#x([0-9a-fA-F]+);（十六进制）
修复
web/src/utils/text.ts：先解 hex → 再解 decimal → 再解 &amp; / &lt; 等命名实体

⑧ KB 状态枚举注释搞反：0/1/2 全部错位

P2
症状
ITHub admin 端状态显示 "Published"，Portal Drawer 显示 "待发布"
根因
代码注释写成 0=Draft, 1=Open/Pending, 2=Published，实际是 0=Draft, 1=Published，没第三态
修复
改 STATUS_LABELS，删掉 status=2 的"已发布"项

⑨ Vite tree-shake 把 BUILD_STAMP 删了，bundle hash 不变

P2
症状
改了前端代码、push 完，浏览器还吃旧 bundle，新功能看不见
根因
顶层加了 const BUILD_STAMP = '…' 但没被引用，被 Vite tree-shake 掉，content-hash 跟着不变
修复
让常量被 console.info(...) 引用，hash 每次必变

09AI Vibe Coding 工作流

这个项目 97 次提交里，没有一行代码是我手写的。从立项到上线，循环就是这 4 步：

STEP 1

Describe intent

用自然语言描述想做什么、看到什么现象、需要修什么。不需要"该怎么实现"的细节。

STEP 2

Claude proposes

AI 读上下文、读相关文件、给方案 + 直接写代码 + 改 build/dev 配置。一次成型，不是草稿。

STEP 3

User verifies

人去浏览器看效果、看 console、看 ITHub admin 后台是不是符合预期。截图 + 报错原文回贴给 AI。

STEP 4

Refine

AI 基于反馈定位根因（不是绕过），重写/调整代码。验证后进 git，进 PR，进 deploy。

4 个具体场景

场景 1 · 探针迭代（KB publish 那一仗）

👤 "ITHub KB 写不进去，PUT 返回 200 但内容是空的"
🤖 加 fillProbeV1，扫 4 种字段命名 → 都没生效
👤 "看 fillProbeV1 的请求 body，它对了吗？"
🤖 加 V2 试 multipart/form-data，V3 试 OData paren path，V4 试嵌套 .Html，V5 修 Content-Type bug
👤 "V5 终于能 PUT 了，但内容还是空，response body 是 true"
🤖 锁定根因：字段名是 Description 不是 DescriptionText，改 /kb/publish 路由，清掉 5 个 fillProbe

场景 2 · 状态枚举修正

👤 [截图] "ITHub 端状态是 Published，Portal 抽屉显示待发布"
🤖 读 STATUS_LABELS，对比注释：1=Open/Pending 是我猜的，实际 ITHub 是 1=Published
👤 "好，改吧，顺便让 Vite 输出新 hash 我才能看到"
🤖 改标签 + bump BUILD_STAMP + push + 等 deploy

场景 3 · 实体解码 + React error #31

👤 [截图] KB 文章正文是 &#x67E5;&#x8BE2;
🤖 改 decodeHtmlEntities，加 hex 解码分支
👤 "现在又点开 KB 整页白屏，console 报 React error #31，对象被当子节点渲染"
🤖 加 safeStr，遇到对象抽 .Name/.UserName/.DisplayName

场景 4 · 缓存刷新

👤 "没看到刷新按钮，怎么没生效？"
🤖 翻部署记录，前一版 bundle 是 Bl1IK8Eo，浏览器在吃旧缓存；新 bundle 是 BA12Td0m 包含刷新代码。判断：CDN 没刷新
👤 "那你让部署每次都出新 hash"
🤖 加 BUILD_STAMP 常量，被 console.info() 引用，hash 必变 + 加 no-cache meta 兜底

这个工作流学到的 3 件事

  1. 先描述现象，不要先描述方案。 "点了按钮页面变白" 比 "用 safeStr 处理一下" 高效 10 倍 —— AI 会自己去查 ITHub 返回什么、React 报什么错、自己定位。
  2. 把环境信息贴全。 console 错误原文、截图、payload 实际值 —— 缺一个就可能猜错。
  3. 探针是工具不是代码。 临时 _debug/ 路由专门为 "我也不知道 ITHub 想要什么" 这种情况服务。跑出来 200 就搬走，探针就丢。

10展望：可扩展方向

4 天做完 MVP，剩下可做的（按 ROI 排序）：

  1. KB 智能推荐：工单创建时基于 Summary 自动推荐 3 篇最相关 KB，agent 转人工时一键附议 —— 减少 50% 重复单
  2. Admin 评分趋势图：当前 AdminStatsWidget 只显示累计数；加一个 30 天 👍/👎 折线图看模型漂移
